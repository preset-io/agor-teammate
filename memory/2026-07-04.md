## a11y: CRON schedule input missing aria-label (10:00 UTC)

**Issue found:** `AlertReportCronScheduler.tsx:108` (`superset-frontend/src/features/alerts/components/`) — the CRON expression text `<Input>` (shown when a user picks "CRON Schedule" in the Alert/Report scheduling modal) had no `aria-label`, `aria-labelledby`, or associated `<label>`. Only a placeholder (`t('CRON expression')`) plus a visually adjacent but structurally unassociated `<div className="control-label">` reading "Schedule". Screen reader users tabbing in heard no accessible name.

WCAG 2.1 SC 1.3.1 (Info and Relationships) / 3.3.2 (Labels or Instructions) — Level A.

Found via an Explore-agent sweep after checking `gh pr list --search "aria-label OR accessibility OR a11y"` (~50 open PRs, incl. #39230-#39245 broad sweeps, #40062, #37805, plus this week's own #41619/#41470/#41742) and ruling out several other candidates already touched by prior sessions (`FilterTitleContainer.tsx`, `ItemTitleContainer.tsx`, `DraggableFilter.tsx`, `VizTypeGallery.tsx` both `<img>` tags already have `alt`, `BaseIcon.tsx` systemic issue flagged out-of-scope). Confirmed no in-flight PR touches `AlertReportCronScheduler.tsx` or the CRON-schedule flow.

**Fix:** added `aria-label={t('Schedule')}` to the `<Input>` — 1-line diff. Matches the naming convention already used by the sibling `Select` above it (`ariaLabel={t('Schedule type')}`, which itself mirrors its own adjacent visible label). Confirmed `Input` is a bare re-export of antd's `Input`, which spreads unrecognized props (incl. `aria-label`) onto the underlying `<input>` DOM element — the prop reaches the DOM correctly. Confirmed single call site (`AlertReportModal.tsx:2674`) via grep.

Commit: `ebdfd2b8858` on `fix/a11y-cron-schedule-input-label` (off `origin/master`, pushed directly to apache/superset — same push-access account as recent days' fixes).

Candidate for a follow-up ticket, not this narrow fix.
