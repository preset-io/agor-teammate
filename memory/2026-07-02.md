## a11y: VizTypeGallery search clear icon aria-label (10:00 UTC)

**Issue found:** `VizTypeGallery` (`superset-frontend/src/explore/components/controls/VizTypeControl/VizTypeGallery.tsx:781`) renders a "clear search" icon (`Icons.CloseOutlined`) with `onClick={stopSearching}` in the search input's suffix, but no explicit `aria-label`. It fell back to the shared `BaseIcon` component's auto-generated label (derived from the icon's component name — `genAriaLabel("CloseOutlined")` → `"close"`), which is generic and doesn't describe what the control does in context (clears the search and returns to the previously active category).

WCAG 2.1 SC 4.1.2 — Level A violation (Name, Role, Value): the accessible name present was non-contextual/misleading, not truly absent.

**Note on the codebase's aria-label mechanism:** `BaseIcon.tsx` auto-generates a generic `aria-label` for every icon based on its component name, and sets `role="button"` when `onClick` is present. This means most "icon-only button missing aria-label" candidates in this repo already have *some* fallback name — the recurring fix pattern (this PR + #41470, #41503, #41619) is replacing that generic auto-label with an explicit, translated, contextual one at the call site. Confirmed via `BaseIcon.tsx` prop-spread order that an explicit `aria-label` passed by the caller overrides the auto-generated one (spread after the default).

**Investigated but ruled out** (to avoid duplicating in-flight work): checked `gh pr list --search "aria-label OR accessibility OR a11y"` (51 open PRs, incl. #39230-#39245 broad WCAG sweeps, #40062 role=button-antipattern) and 3 candidates from a broader sweep — `ControlHeader.tsx:103` (info icon) and `TableCatalog.tsx:71` (table-catalog remove icon) were also clean and unclaimed, picked `VizTypeGallery.tsx` as the smallest, most isolated single-call-site fix.

**Fix:** `aria-label={t('Clear search')}` on the icon — 3 line insertion. Commit `1aa99dbb6ae`. Only call site of this icon/handler pattern in the codebase (confirmed via self-review grep).

## Deprecation-warning fix: sqlalchemy-redshift pkg_resources UserWarning (16:00 UTC)

**Warning found:** first result of Datadog `deprecat*` search (`now-1d`→`now`): `/usr/local/lib/python3.10/site-packages/sqlalchemy_redshift/__init__.py:1: UserWarning: pkg_resources is deprecated as an API...` — firing repeatedly (14+ occurrences/day) on both `web` and `worker` pods whenever a Redshift connection is established. This is the same warning ruled out on 2026-07-01 as "third-party, outside apache/superset" — today I dug deeper instead of re-dismissing it, since the task explicitly said to pick the first result.

**Why the obvious fix (bump the pin) doesn't work:** `sqlalchemy_redshift`'s own `__init__.py` imports `pkg_resources`. Only the `1.0.0` release removes this (switched to `importlib.metadata`), but `1.0.0` requires `SQLAlchemy>=2.0.0,<3`, which conflicts with `apache/superset`'s `sqlalchemy>=1.4,<2` pin (`pyproject.toml:110`). Confirmed via `gh`: **PR #39750** (Dependabot bump of the `sqlalchemy-redshift` pin to allow `1.0.0`) was closed by maintainer **rusackas**: *"Closing... let's wait for SQLAlchemy 2."* — an explicit, authoritative "not yet" from upstream. Also found **open issue #36082** ("[Redshift][BUG] sqlalchemy-redshift error due to deprecated setuptools package") tracking this exact warning, unaddressed — this PR closes it.

**Precedent found:** `superset/mcp_service/__init__.py` already suppresses an analogous unavoidable third-party warning (`authlib.jose`, from bundled `fastmcp`) via `warnings.filterwarnings(...)`, merged in **PR #40977**. Same shape of problem (deprecated call inside a dependency we can't fix or upgrade past), same remedy.

**Fix:** added `warnings.filterwarnings("ignore", message=r"pkg_resources is deprecated as an API")` to `superset/db_engine_specs/redshift.py`. Commit `5569a6cca45`, corrected in `a6ab24ce80b` (see self-review below). No `category=` set deliberately — verified `requirements/base.txt` pins `setuptools==80.9.0`, which raises this as plain `UserWarning`, not `DeprecationWarning`; adding the category would silently break the match.

### #41263 and #41286 — approved but CI-blocked, root-caused and fixed

Both were `mergeStateStatus: BLOCKED`. Checked whether `Python-Integration`/`Python-Unit` were also failing on `master` (they weren't — both green on master) — so these were **not** pre-existing noise, ruling out the "re-run and wait" path multiple prior sessions took on #41263 (2026-06-25 through 2026-06-29, always re-ran and it kept failing — that's because it wasn't noise).

**#41263 root cause:** the PR's own fix (`if old in data: data.setdefault(new, data.pop(old))`) solved the original falsy-value bug (`timeseries_limit=0` not renaming) but silently changed merge semantics for `groupby`→`columns`: previously `groupby` always *overwrote* `columns` when both were present; `setdefault` now makes it a no-op whenever `columns` already has a value. This broke `test_time_offset_comparison_queries_use_chart_row_limit`/`..._without_chart_limit` (KeyError: "None of ['name'] are in the columns") and directly contradicts the repo's own unit test `test_chart_data_query_object_schema_deprecated_fields_renamed`, which explicitly asserts "groupby overwrites columns when both are provided" (that unit test was in fact the one failing on CI: `assert ['country_name'] == ['region']`).

Fix: restored overwrite semantics for all four renames, but treat `value == 0` as significant (not just truthy) so `timeseries_limit=0` still renames — satisfies both the original bug fix intent and every existing test case (verified by hand against all 8 assertions in the unit test plus the two integration-test scenarios). Commit `d2b6561e20` on `fix/query-object-schema-rename-falsy`.

**#41286 root cause:** the PR's own newly-added test (`test_deprecated_logs_warning_exactly_once`) asserts `"5.0.0" in mock_logger.warning.call_args[0][0]` — but `logger.warning(message, *args)` uses lazy %-style formatting, so `call_args[0][0]` is the raw un-interpolated template (`"%s.%s ... version %s"`), which never contains `"5.0.0"`. The application logic (once-per-endpoint-per-process warning) is correct and already approved; only the test's own assertion was wrong. Fixed to check `"5.0.0" in mock_logger.warning.call_args[0]` (the full arg tuple). Commit `875e0195af` on `fix/deprecated-decorator-warn-once`.

First push attempt for #41263 went to `apache/superset` by mistake (created a stray unrelated branch there, since `git push origin` defaulted to the `origin` remote which is `apache/superset`) — caught it immediately (PR's `head.sha` didn't change), deleted the stray branch, and re-pushed to the correct fork URL.
