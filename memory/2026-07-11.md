**Search:** `gh pr list --repo apache/superset --state open --search "aria-label OR accessibility OR a11y"` turned up ~20 open a11y PRs (the WCAG-series #39230–#39245, plus #41470/#41619/#41903/#41931/#40062/#28171 from recent daily sweeps). **Found:** `superset-frontend/src/components/CopyToClipboard/index.tsx` — `getDecoratedCopyNode()`'s `cloneElement` branch (used whenever a caller passes a custom `copyNode` React element) wired `onClick` but had **no `onKeyDown` handler at all**, and left `tabIndex` as whatever the caller passed (usually `undefined`). Result: custom `copyNode` elements that already look accessible — e.g. `explore/components/EmbedCodeContent.tsx`'s `<span role="button" aria-label={t('Copy to clipboard')}><Icons.CopyOutlined /></span>` — were not keyboard-focusable or keyboard-operable at all. The sibling default-`<span>` branch a few lines below already had the correct `tabIndex={0}` + Enter/Space `onKeyDown` pattern; this was a latent gap only in the `cloneElement` path. Confirmed via `gh pr diff`/`gh pr list --search` on the filename that none of the ~20 open a11y PRs touch this file. WCAG 2.1 SC 2.1.1 (Keyboard) — Level A.

**Fix (1 file + 1 test file, root-cause/shared-component):** extended the `cloneElement` branch with the same `onKeyDown` handler (Enter/Space → `onClick`, `preventDefault()`, respects `disabled`) and changed `tabIndex: disabled ? -1 : node.props.tabIndex` → `tabIndex: disabled ? -1 : (node.props.tabIndex ?? 0)`. Single shared-component fix closes the gap for every call site passing a custom `copyNode` (`EmbedCodeContent.tsx`, `URLShortLinkButton/index.tsx`, `TablePreview/index.tsx`, etc.) without touching each call site.

**Branch/commit:** `fix/a11y-copytoclipboard-keyboard`, off `origin/master`, commit `69876ec546` (amended in a review-feedback test after self-review — no push had happened yet, so amend was safe per [[feedback_no_force_push_on_pr_updates]], which only bars amend+force-push on *already-open* PRs).

Worth remembering: a confident, well-argued subagent verdict was empirically wrong once checked against the real CI execution mode (full test file), not just an isolated single-test repro — always verify "required change" findings before applying them, especially anything involving import-order-dependent global state like `warnings.filters`.
