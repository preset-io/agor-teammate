## Root cause investigation

The warning fires from `QueryObject._rename_deprecated_fields()`. Traced the source:

`ChartDataQueryObjectSchema.rename_deprecated_fields()` (`superset/charts/schemas.py:1469`) is a Marshmallow `@post_load` that should rename deprecated fields before they reach `QueryObject.__init__`. But the implementation had a bug:

```python
# old (buggy)
if value := data.pop(old, None):
    data[new] = value
```

`data.pop(old, None)` always removes the old key. But when `value` is falsy — `timeseries_limit=0` (the default for most charts), `groupby=[]`, `None` values — the truthiness check skips setting `data[new]`. So the new key (`series_limit`) is never added to the dict. `QueryObject` falls back to its `series_limit: int = 0` default — numerically correct but semantically wrong (caller's explicit 0 is silently discarded).

Note: the walrus bug does NOT cause the `_rename_deprecated_fields` warning to fire via the schema path (the old key is always popped). The warning fires from callers that bypass the schema (e.g. `/api/v1/query/` endpoint). The fix is still correct: proper rename semantics ensure `series_limit` is explicitly set when `timeseries_limit` is provided.

```python
# new (correct)
for old, new in _renames:
    if old in data:
        data.setdefault(new, data.pop(old))
```

- `old in data` — presence check, not truthiness
- `data.pop(old)` — always removes deprecated key
- `data.setdefault(new, ...)` — sets new key only if not already present (explicit modern key wins)
