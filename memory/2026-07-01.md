## a11y: cross-filter search icon aria-label (10:00 UTC)

**Issue found:** `CrossFilterChartTitle` (`superset-frontend/src/dashboard/components/nativeFilters/FilterBar/CrossFilters/CrossFilterTitle.tsx`) renders a search icon (`StyledIconSearch`, wraps `Icons.SearchOutlined`) with `role="button"`, `tabIndex={0}`, and `onClick={onHighlightFilterSource}` but no `aria-label` — screen readers would announce only the raw SVG. It's visually wrapped in `<Tooltip title={t('Locate the chart')}>` but that text isn't exposed as an accessible name.

WCAG 2.1 SC 4.1.2 — Level A violation (Name, Role, Value).

**Before picking this issue, checked:**
- memory/2026-06-27, 2026-06-28: fixed IconTooltip and ActionButton — this file untouched
- 17 in-flight a11y PRs on apache/superset (`gh pr list --search "aria-label OR accessibility OR a11y"`) — checked file lists of the most relevant ones (#39230, #40062, #41470, plus WCAG-specific series #39230-39245, #28171, #39863) — none touch `CrossFilterTitle.tsx`
- Confirmed single call site (`CrossFilter.tsx:76`) via grep — no other places using this pattern were missed

**Fix:** `aria-label={t('Locate the chart')}` — reuses the same translated string already shown in the tooltip. Commit `244dcc65cd8`.

**Pre-commit note:** Type-checking failed on an unrelated file (`TextAreaControl.tsx`) due to a stale built `.d.ts` in `packages/superset-ui-core/lib/components/ModalTrigger/index.d.ts` (declared `modalTitle?: string` when the source already has `modalTitle?: ReactNode`). Patched the `.d.ts` locally (not committed — it's a generated artifact) to unblock the hook, per standing guidance.

## Deprecation-warning sweep (16:15 UTC) — no PR opened

**Task:** find one deprecation warning in prod Datadog logs, fix it, PR, self-review, EngCodeReviewBot, Shortcut. Result: investigated three candidates in depth, all dead ends — no PR opened today. Documenting the investigation since it surfaced a real operational finding.

**Datadog query:** `deprecat*`, `now-1d` to `now`, first result:
```
appbuilder.app is deprecated and will be removed in a future version. Use current_app instead
(base.py:234, funcName=app, service=superset, env=production, image=preset/superset-private@ee43906d)
```

**Candidate 1 — `appbuilder.app` (the literal first result):**
Traced `base.py:234` to `flask_appbuilder`'s own `AppBuilder.app` property getter (confirmed by reading the installed package). Exhaustively grepped `apache/superset` for `appbuilder.app`, `builder.app`, Jinja templates, MCP service code — **zero live call sites**. `gh search prs "appbuilder.app" --repo apache/superset` turned up **PR #40876** ("fix: replace deprecated appbuilder.app with current_app"), merged **2026-06-09**, which removed the last 2 usages (`superset/initialization/__init__.py`, `spa.html`). Confirmed clean on current `origin/master`. This warning is not fixable via a new apache/superset PR — it's either the FAB library's own internal legacy SQLAlchemy session shim (unrelated `Session.app` attribute, not `AppBuilder.app`) or code in the proprietary `superset-private` fork, which I don't have access to from this workspace.

**Candidate 2 — `groupby`/`timeseries_limit` deprecated field warnings** (`query_object.py:223`, also showed up in the last 24h):
Traced to `QueryObject._rename_deprecated_fields`. Found the only non-test constructor of `QueryObject` outside the class itself is `QueryObjectFactory.create()` (`superset/common/query_object_factory.py`), which **already strips and renames all `DEPRECATED_FIELDS` before constructing `QueryObject`** — via **PR #41204** ("fix(query_object_factory): normalize deprecated query fields before constructing QueryObject"), merged **2026-06-18**. That PR's own description confirms it was opened for this *exact* Datadog log line (root cause: `Slice.get_query_context()` loading old stored `query_context` JSON with `groupby` still set, on every render). Already fixed upstream.

**Candidate 3 — `Superset.explore_json` deprecated-endpoint warning** (`views/base.py:186`, `@deprecated` decorator):
This one *is* real and *is* superset's own code (`superset/views/core.py:297`), but it's intentionally still hit by any viz plugin not yet migrated to the v1 Chart Data API — gated by `useLegacyApi` in `getChartMetadataRegistry()` (`superset-frontend/src/explore/exploreUtils/index.ts`). This is a deliberate, registry-driven, multi-year plugin migration (SIP-scale), not a bug with a minimal safe fix.

**Also checked and ruled out:**
- `authlib.jose` deprecation (`preset/preset/mcp/oauth_factory.py`) — lives in `superset-private`, not `apache/superset`. Local git stash history in this worktree shows a fix for this exact warning (`aminghadersohi/fix-bigquery-schema-default-dataset` stash referencing "migrate MCP service JWT errors from authlib.jose to joserfc (#40582)") was already attempted **and reverted** (`revert-joserfc-jwt-errors` stash).
