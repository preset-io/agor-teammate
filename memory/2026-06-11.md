## sc-108451: selectFirst + dependent filter race condition

**Bug:** Dashboard filters where both A and B have `defaultToFirstItem = true` and B depends on A apply the wrong first value to charts on load.

**Linked to:** sc-102912 (original fix: PR #39504, merged 2026-04-22)

**Root cause:** Readiness guard in `FilterValue.tsx` — both `selectedParentFilterValueCounts` and `depsCount` are 0 on first load, so 0 === 0 lets B fetch unfiltered data immediately. B auto-selects wrong first item and dispatches (first-initialization via `requiredFirst`). When A selects later, B re-selects correct value but it's no longer first-initialization so `shouldDispatch = false`. Chart stuck with B's wrong value.

**Fix:** In readiness guard, before the count comparison, block B if any transitive parent with `defaultToFirstItem = true` has `filterState.value === undefined`.

## sc-108187: "Overwrite editor" in Query History doesn't replace SQL text

**Story:** https://app.shortcut.com/preset/story/108187  
**Reporter:** Rebeca Nogueira  
**Status:** → Reviewing (PR open)

### Root Cause

When `SqllabBackendPersistence` is enabled, queries store `sqlEditorId = qe.tabViewId` (the backend-assigned ID, e.g. `"42"`) rather than the client-side nanoid (e.g. `"ABCdef123"`). The `restoreSql` handler in `QueryTable` dispatches `queryEditorSetSql({ id: query.sqlEditorId }, sql)` — so `action.queryEditor.id = tabViewId`.

The `QUERY_EDITOR_SET_SQL` reducer calls `alterUnsavedQueryEditorState(state, update, tabViewId)`. That function checks `tabHistory[last] !== tabViewId` — TRUE because `tabHistory` tracks client-side IDs. So it routes to `alterInArr(queryEditors, { id: tabViewId }, ...)` looking for an editor with `id === tabViewId`. No match (editors have client-side IDs). State unchanged. Editor never updates.

**Fix:** Added same `tabViewId → client-side id` normalization in `QUERY_EDITOR_SET_SQL` that `START_QUERY` already uses — `getFromArr(queryEditors, tabViewId, 'tabViewId')?.id ?? tabViewId`.

### Files Changed

- `superset-frontend/src/SqlLab/reducers/sqlLab.ts` — normalize ID in `QUERY_EDITOR_SET_SQL`
- `superset-frontend/src/SqlLab/reducers/sqlLab.test.ts` — new test covering tabViewId path

## sc-108475: Native filter default value silently fails to persist when values are created/pasted

**Story:** https://app.shortcut.com/preset/story/108475
**Reported:** 2026-06-10 by Jason Araos Fuentes
**SC state:** → Implementing (500020185), Epic 101517

### Bug
Value-type native filter with "Allow creation of new values" enabled — pasting comma-separated values as default (e.g. `10107, 10121, 10134, ...`) shows chips in the config modal but they're NOT persisted after Save. Filter shows empty on dashboard; re-opening shows no default value.

### Root Cause (3 compounding bugs)

1. **PRIMARY — Paste tokenization doesn't trim whitespace** (`Select.tsx` ~line 693):
   `"10107, 10121".split(",")` → `["10107", " 10121"]` — leading space on every value after first comma. Space prevents label matching → values stored as wrong string literals with spaces.

2. **SECONDARY — Multi-select never adds typed values to options** (`SelectFilterPlugin.tsx` ~line 326):
   `!multiSelect` guard in `options` useMemo blocks typed/created values from appearing as selectable options for multi-select filters.

3. **TERTIARY — `uniqueOptions` doesn't include already-selected created values** (`SelectFilterPlugin.tsx` ~line 316):
   `uniqueOptions` built only from DB data. Created values in `filterState.value` not present in DB aren't in the options list.
