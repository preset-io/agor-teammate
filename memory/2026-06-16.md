## sc-111015: Native ClickHouse JSON and Map type columns do not render JSON cell viewer in SQL Lab

**Story:** [SC-111015](https://app.shortcut.com/preset/story/111015)
**SC state:** → Implementing (500020185)
**Epic:** 101517 (Stabilize Master)

### Root Cause

ClickHouse native JSON/Map types return as Python `dict` objects from clickhouse-connect. Superset's result serialization converts them via `str()` → single-quoted Python repr (e.g. `{'a': 1}`) → `JSON.parse()` fails on frontend → no cell viewer. String cast columns produce valid double-quoted JSON and work correctly.

### Fix Target

Generic path in result serialization: `SupersetJSONEncoder` in `superset/utils/core.py` or `SupersetResultSet` in `superset/result_set.py`. When a cell value is a `dict` or `list`, use `json.dumps()` not `str()`.

## sc-108003: Legend margin cursor steals focus to Results search

**Story:** [SC-108003](https://app.shortcut.com/preset/story/108003)
**SC state:** → Implementing (500020185)
**Epic:** 101517 (Stabilize Master)
**Priority:** P3 | **Customer:** Ribbon (PCS)

### Root Cause

`FilterInput` in the Results pane (`DataTableControls.tsx:96`) has `shouldFocus` hardcoded to `true`. On mount, it calls `inputRef.current.focus()`. When `legendMargin` (renderTrigger control) changes, it changes `queryFormData` → `cappedFormData` → WeakMap cache miss in `useResultsPane` → `setIsLoading(true)` → `SingleQueryResultPane` unmounts → remounts after fetch → `FilterInput` re-mounts and steals focus from the legend margin input.

### Fix Target

`superset-frontend/src/explore/components/DataTableControl/index.tsx` — guard the `useEffect` in `FilterInput` to skip auto-focus when another input/textarea/contenteditable already has focus.

## sc-107864: Chart context link in dashboard comment panel not visible to all users

**Story:** [SC-107864](https://app.shortcut.com/preset/story/107864)
**SC state:** → Implementing (500020185)
**Epic:** 101517 (Stabilize Master)
**Repo:** superset-shell (Preset-only, formerly "Cord" feature)

### Root Cause

`ThreadSource.tsx` and `ThreadOverride.tsx` call `thread.useThread(threadId)` from the Cord SDK to get `metadata.chartId`. For non-authors viewing threads via the dashboard panel (`partialMatch: true`), the Cord SDK does not return full thread metadata — only the thread author gets it back. So `chartId` is undefined for non-authors, and they see "Dashboard thread (global)" or no header at all instead of the chart name.

### Fix

Create a `ThreadsContext.tsx` React context that holds a `threadId → chartId` map. Populate it in `ThreadPopoverContent` from its existing `useThreads` call (which works for ALL users). Consume the context in `ThreadSource` and `ThreadOverride` as a fallback when `thread.useThread` doesn't return metadata.

Files changed:
- `superset-frontend-overrides/preset-extensions/cord/ThreadsContext.tsx` (new)
- `ThreadPopoverContent.tsx` — populate context, wrap return in provider
- `ThreadOverride.tsx` — use context as fallback, relax null guard
- `ThreadSource.tsx` — use context chartId as fallback
- `ThreadSource.test.tsx` — add test for non-author case
