## ~23:50 UTC ‚Äî Dashboard screenshot timeout investigation & fix

### Investigation summary

Investigated report failures for workspace `990e161d`. Root cause confirmed: tab-specific dashboard report schedules time out waiting for `<body class="standalone">` in Playwright.

**Call chain:**
1. `AsyncExecuteReportScheduleCommand.run()` ‚Üí `@transaction()` ‚Üí sets `g.in_transaction = True`
2. `BaseReportState.next()` ‚Üí `update_report_schedule_and_log(WORKING)` ‚Üí `create_log()` ‚Üí `db.session.commit()` (direct, bypasses nesting guard) ‚Äî WORKING state committed
3. `send()` ‚Üí `_get_notification_content()` ‚Üí `_get_screenshots()` ‚Üí `get_dashboard_urls()` ‚Üí `_get_tab_url()`
4. `CreateDashboardPermalinkCommand.run()` ‚Üí `@transaction()` nesting guard fires ‚Üí only `db.session.flush()`, no commit
5. Playwright navigates to `/dashboard/p/<key>/` ‚Äî Flask 404 (row not in DB for other connections) ‚Üí no `<body class="standalone">` ‚Üí 600 s timeout

**Why Option 3 is safe here:** `create_log()` (line 196) already does a direct `db.session.commit()` with pylint disable before `_get_screenshots()` is called. At the point `_get_tab_url()` runs, the only uncommitted write is the permalink INSERT. This is the same pattern the codebase already uses.

### Fix

**File:** `superset/commands/report/execute.py`, `BaseReportState._get_tab_url()` (~line 360)  
**Branch:** `fix-report-tab-permalink-commit` in apache/superset (off master, from `alerts-reports-working-timeout-sc-104379` worktree)  
**Commit:** `9058b49971`

Added 4 lines after `CreateDashboardPermalinkCommand(...).run()`:
```python
# Commit the permalink immediately so Playwright's separate DB connection
# can resolve the URL. CreateDashboardPermalinkCommand only flushes when
# called inside an outer @transaction(), leaving the row invisible to
# other connections until we explicitly commit here.
db.session.commit()  # pylint: disable=consider-using-transaction
```
