## a11y: ViewportControl numeric inputs missing aria-label (10:00 UTC)

**Issue found:** `superset-frontend/src/explore/components/controls/ViewportControl.tsx` — the deck.gl map "Fixed viewport" popover control renders 5 numeric `<TextControl>` inputs (longitude, latitude, zoom, bearing, pitch), each with a visible sibling `<FormLabel>{ctrl}</FormLabel>`, but never passed a `label` prop into `TextControl`, so `TextControl`'s underlying `<Input>` (which only sets `aria-label={props.label}`) got no accessible name at all. All 5 inputs were completely unlabeled to assistive tech.

WCAG 2.1 SC 4.1.2 (Name, Role, Value) — Level A.

**Search process:** Given the unusually large amount of a11y work already in flight on this repo (`gh pr list --search "aria-label OR accessibility OR a11y"` returned 29 open PRs, including a systematic WCAG 1.1.1–3.3.1 series #39230–#39245, the role=button-anti-pattern PR #40062, plus #41903/#41470/#41619/#28171 from prior sweeps), delegated the search to an Explore agent with the full exclusion list (files + PR numbers + WCAG categories already covered). It ruled out several candidates (icon-only `Button` instances — already have auto-generated aria-labels via `BaseIcon.tsx`'s `genAriaLabel()`; `TextAreaControl.jsx`'s unlabeled branch — currently unreachable in production, every call site sets `language`) before landing on `ViewportControl.tsx`, cross-referenced against all 29 PRs' file lists (277 unique paths) — no overlap.

**Fix (2 commits, see below):**
1. First commit: added `label={ctrl}` directly to the `<TextControl>` call — mechanically correct (aria-label reached the DOM) but not verified against the full component tree.
2. Self-review subsession caught a real regression: `TextControl` unconditionally spreads `label` into `ControlHeader`, whose only guard against rendering is `if (!label) return null`. Passing `label={ctrl}` therefore made `ControlHeader` also render its own visible `FormLabel`, duplicating each input's visible text ("longitude" appeared twice in the DOM — confirmed via instrumented render test). Not an established pattern elsewhere in the file/dir — a genuine bug introduced by the first commit.
3. Second commit fixed it properly: added a dedicated `ariaLabel` prop to `TextControl` (`superset-frontend/src/explore/components/controls/TextControl/index.tsx`) that flows straight to `<Input aria-label={ariaLabel ?? label}>` without being forwarded to `ControlHeader`, and switched `ViewportControl` to use `ariaLabel={ctrl}` instead of `label={ctrl}`. Purely additive/optional prop — grepped all other `TextControl` call sites (`jsonFormsHelpers.tsx`, `explore/controls.tsx`, `ContourPopoverControl.tsx`, `FixedOrMetricControl/index.tsx`, `sections.tsx`, `AnnotationLayer.tsx`, `DatasourceEditor.tsx`) — none pass `ariaLabel`, so none are affected. Verified via a scratch render test (not committed): exactly 1 visible "longitude" text node post-fix, and `aria-label="longitude"`/`"latitude"` correctly present on the first two inputs. `ViewportControl.test.tsx` (existing suite) still passes.

**Pre-commit note:** both commits passed pre-commit clean (type-check, prettier, oxlint, custom rules) — no stale `.d.ts` patching needed this time.

**Commits:** `f2c3bb870b` (initial fix) + `b64a798b46` (regression fix after self-review) on `fix/a11y-viewport-control`, off `origin/master`, pushed to apache/superset.

**Picked:** `pkg_resources is deprecated as an API` (UserWarning from sqlalchemy-redshift's own `__init__.py:1`, since it still does `from pkg_resources import ...`), 76/day (61 py3.10 web/celery + 15 py3.11 mcp_service). Not in `release-health-state.json`'s `known_noise` list yet — genuinely unaddressed. Real bonus: there's already an *attempted* fix for this exact warning in `superset/db_engine_specs/redshift.py` (a `warnings.filterwarnings("ignore", ...)` with a comment reasoning it's registered early enough via `Database.db_engine_spec` access before `create_engine()` triggers SQLAlchemy's lazy dialect import) — yet it's still firing at volume in prod. Traced the ordering logic in `Database._get_sqla_engine()` and `load_engine_specs()` at length; couldn't conclusively prove the exact root cause for the web-process share without live production debugging (likely a proprietary/downstream bootstrap path outside this OSS repo racing to import `sqlalchemy_redshift` first). Confirmed empirically (PyPI wheel inspection) that `sqlalchemy-redshift` 1.0.0 fixed this upstream (switched to `importlib.metadata`) but Superset's pin (`<0.9`, per `pyproject.toml`) can't move there without a SQLAlchemy 2.0 migration (per the existing comment, referencing apache/superset#39750) — not attempting that bump today, too risky/out of scope.

**Fix (3 files):**
1. `superset/mcp_service/__init__.py` + `superset/mcp_service/server.py`: added the same filter, mirroring the *exact* existing pattern already used for `authlib.jose` in these same two files. mcp_service is a separate process/import graph from the main app and had zero suppression for this specific warning — a clean, provable gap (confirmed via grep: no `pkg_resources` filter existed there before).
2. `superset/utils/logging_configurator.py`: registered the same filter unconditionally in `DefaultLoggingConfigurator.configure_logging()`, which runs first in `SupersetAppInitializer.init_app()` (before `check_and_warn_database_connection()`, `configure_celery()`, etc.) — a more robust, order-independent primary suppression point for both the web app and celery workers, since it doesn't depend on `Database.db_engine_spec` having been touched first.
3. `superset/db_engine_specs/redshift.py`: left the original filter in place (idempotent, harmless), updated its comment to point at the new primary suppression location.
