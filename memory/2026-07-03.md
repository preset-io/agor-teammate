## a11y: FilterTitleContainer/ItemTitleContainer non-functional `alt` prop (10:00 UTC)

**Issue found:** three call sites in `superset-frontend/src/dashboard/components/nativeFilters/FiltersConfigModal/` (`FilterTitleContainer.tsx:214`, `ItemTitleContainer.tsx:182`, `DraggableFilter.tsx:99`) passed an `alt` prop to `Icons.*` components intending to set an accessible name. `alt` isn't a recognized prop on these components (built on shared `BaseIcon.tsx`) — it's just spread onto the DOM as an inert attribute — so screen readers fell back to `BaseIcon`'s generic auto-generated label ("delete", "drag") instead of the intended contextual one. This is a *different* flavor of bug than prior days' fixes (#41681, #41619, #41470): not a missing label, but a label that looks present and intentional in the source yet is silently non-functional due to a wrong prop name.

WCAG 2.1 SC 4.1.2 — Level A violation (Name, Role, Value).

Checked `gh pr list --search "aria-label OR accessibility OR a11y"` (~20 open a11y PRs, including the broad #39230-#39245 WCAG sweeps and #40062 role=button replacement) and diffed #41470/#41619/#39230 directly — none touch these three files.

**Fix (as originally scoped):** rename `alt=` → `aria-label=` at all three call sites. 3-line diff. Committed as `5569a...` (superseded, see self-review).

It contains a reviewed, LGTM'd (3x Claude review + human review), zero-behavior-change fix: wraps the two `authlib.jose` imports in `warnings.catch_warnings()` + `filterwarnings("ignore", message=r"authlib\.jose module is deprecated")`.    - It stalled on an explicit, never-answered team-alignment question Amin raised on 2026-05-27: code-level suppression (this PR) vs. a Datadog index-level exclusion filter. 3. **The real root-cause fix (migrate `authlib.jose` → `joserfc`) is genuinely riskier than a same-day SRE fix.** `oauth_factory.py:502` calls `JsonWebToken(["HS256"]).decode(...)` directly and has hand-written comments (lines 490-502) about relying on the precise exception hierarchy (`ExpiredTokenError` vs generic `JoseError`) for token-expiry vs invalid-token branching in an OAuth verification path. There *is* a merged precedent — [apache/superset#40582](https://github.com/apache/superset/pull/40582) migrated `superset/mcp_service/jwt_verifier.py` from `authlib.jose.errors` to `joserfc.errors` — but that file delegates actual decoding to fastmcp's `JWTVerifier` and only had to swap error-class imports; it never had to touch a raw `.decode()` call. Migrating `oauth_factory.py` for real means verifying joserfc's decode raises equivalent errors for the expiry-vs-invalid distinction the code depends on — that's a dedicated, tested piece of work, not a warning fix.
