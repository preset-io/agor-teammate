## a11y: FilterTitleContainer/ItemTitleContainer non-functional `alt` prop (10:00 UTC)

**Issue found:** three call sites in `superset-frontend/src/dashboard/components/nativeFilters/FiltersConfigModal/` (`FilterTitleContainer.tsx:214`, `ItemTitleContainer.tsx:182`, `DraggableFilter.tsx:99`) passed an `alt` prop to `Icons.*` components intending to set an accessible name. `alt` isn't a recognized prop on these components (built on shared `BaseIcon.tsx`) — it's just spread onto the DOM as an inert attribute — so screen readers fell back to `BaseIcon`'s generic auto-generated label ("delete", "drag") instead of the intended contextual one. This is a *different* flavor of bug than prior days' fixes (#41681, #41619, #41470): not a missing label, but a label that looks present and intentional in the source yet is silently non-functional due to a wrong prop name.

WCAG 2.1 SC 4.1.2 — Level A violation (Name, Role, Value).

Checked `gh pr list --search "aria-label OR accessibility OR a11y"` (~20 open a11y PRs, including the broad #39230-#39245 WCAG sweeps and #40062 role=button replacement) and diffed #41470/#41619/#39230 directly — none touch these three files.

**Fix (as originally scoped):** rename `alt=` → `aria-label=` at all three call sites. 3-line diff. Committed as `5569a...` (superseded, see self-review).
